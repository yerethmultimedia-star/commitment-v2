# Event Store & Read Models

## Current State

**Correction to the brief this iteration started from: the event store is not dead code system-wide.** It is dead code for `Task` specifically, but genuinely alive and load-bearing for `Goal`.

Verified directly (not inferred):

- `apps/backend/src/infrastructure/event-store/in-memory-event-store.ts` implements `EventStore` (`@commitment/domain`) with `saveEvents(streamId, expectedVersion, events)` and `getEvents(streamId, fromVersion?)`. `saveEvents` **does** enforce optimistic concurrency: if `currentVersion !== expectedVersion` it throws `ConflictException` (lines 16-20). This is a real, correct implementation, not a stub.
- It is registered as the `'EventStore'` DI token in **two** modules: `task.module.ts:117-118` and `goal.module.ts:53-54`.
- For **Task**: zero call sites of `.saveEvents(`/`.getEvents(` anywhere in `apps/backend/src/task/**`. The registration is genuinely unused — confirms the original "dead code" finding, but only for this one module.
- For **Goal**: 8 real call sites of `eventStore.saveEvents(...)` — `register-goal.handler.ts:53`, `activate-goal.handler.ts:72`, `complete-goal.handler.ts:66`, `rename-goal.handler.ts:66`, `archive-goal.handler.ts:39`, `update-goal-description.handler.ts:70`, `link-commitment-to-goal.handler.ts:67`, `link-habit-to-goal.handler.ts:67` — plus a real read: `get-goal-history.handler.ts:35` calls `eventStore.getEvents(query.goalId)`, feeding a real, routed endpoint: `goals.controller.ts:124` — `@Get(':id/history')`.
- `activate-goal.handler.ts`'s own comment states the design intent explicitly and correctly (lines 69-70): _"Durable event log (ADR-021) — additive, does not affect step 3's source of truth."_ This is **not** Event Sourcing (aggregate state is never reconstructed from these events); it's a parallel, deliberately-additive audit trail. `getEvents` is only ever called by `get-goal-history.handler.ts`, never by anything that loads/rehydrates a `Goal` for a write — confirmed by grep, `applyEvent` on `Goal` is invoked only via `recordEvent` at write time, never during `findById`.
- **Neither `Commitment` nor `Habit` register or use `EventStore` at all** — no history feature exists for either. This is an inconsistency: Goal has a real activity-history endpoint; its sibling aggregates in the same domain don't, with no documented reason found for the asymmetry.
- **The actual source of truth is flat, versioned in-memory state**, not the event log. `InMemoryGoalRepository` (`goal/infrastructure/in-memory-goal.repository.ts`) and `InMemoryCommitmentRepository` (`commitment/infrastructure/in-memory-commitment.repository.ts`) are structurally identical: a `Map<string, Aggregate>` holding **live object references** (not serialized snapshots) plus a `Map<string, number>` version counter that increments by event count on save. **Critically, `save()` in both never checks the current version against an expected one — there is no compare-and-swap, no conflict detection, at the actual state-repository level.** The only place optimistic concurrency is enforced is in the (Goal-only, additive, non-authoritative) event store. A concurrent double-write to the real state would silently last-write-win.
- `apps/backend/src/infrastructure/projections/in-memory-projection.repository.ts` (`InMemoryProjectionRepository<TProjection>`) is a generic key-value store (`save`/`findById`/`findAll`/`delete`). Grep found it's a generic class available for use, but this iteration did not exhaustively trace every consumer across all 4 aggregate modules to confirm which, if any, currently instantiate it versus using ad hoc arrays elsewhere — flagged as unconfirmed rather than guessed at.

## Strengths

- The one real event-store use case (Goal history) is **correctly designed**, not just present: additive audit log, explicit non-authoritative status stated in code comments, real optimistic-concurrency check on the log itself, clean read path via a dedicated query handler. This is exactly right for what it's trying to be.
- `InMemoryEventStore.saveEvents` computing `eventVersion` per-event and stamping `recordedAt` server-side (not trusting the aggregate) is a small but correct detail — it means the log's ordering/timestamps can't be spoofed by whatever called `record`.
- The team's own comment (`activate-goal.handler.ts:69`) shows they know exactly what pattern they built and why — this isn't accidental architecture, it's a documented, deliberate choice (ADR-021), and the code matches the ADR.

## Weaknesses

- **No real optimistic concurrency protection on the actual source of truth.** `InMemoryGoalRepository.save()` / `InMemoryCommitmentRepository.save()` (and, by the same pattern, presumably Task/Habit's) never compare an expected version before overwriting. "Versioned" repository is a misnomer today — it tracks a version number but doesn't enforce it. The only enforced version check lives in a store that isn't the source of truth.
- **The durable-event-log pattern exists for exactly one of four sibling aggregates** (Goal only), with no stated reason why Commitment/Habit don't get the same activity-history capability. Either this is a real product gap (users can't see Commitment/Habit history) or an unstated decision that should be recorded.
- **Task's `EventStore` DI registration is genuinely dead** — confirms part of the original finding. Dead DI wiring is a minor but real smell: it implies either abandoned work or a misleading signal to a future engineer that Task has event logging when it doesn't.
- **No serialization story exists anywhere.** Both the versioned repositories and the event store's in-memory `Map`s hold live JS objects / already-constructed event objects — nothing in this codebase has ever had to answer "how do we turn a `Goal` or a `DomainEvent` into bytes and back." This means Ports & Adapters' "just swap the adapter" story (raised by Iteration 2) is optimistic specifically at this layer: the port signatures are fine, but a real Postgres adapter would be the first thing in the entire codebase to require real (de)serialization, and that work has zero precedent to build on.

## Risks

- **Silent lost updates.** Two concurrent commands against the same Goal/Commitment (e.g., two rapid taps, or a retried request) would both succeed and the second `save()` would silently overwrite the first — no error, no conflict signal, because the state repository doesn't check versions. Low likelihood today (single demo user, sequential usage) but this is exactly the kind of gap that only surfaces under real concurrent load, i.e., precisely when it's most expensive to discover.
- **Documentation actively misdescribes the write-side model.** `docs/02-domain/postgresql_physical_model.md` (§2.A, "CAPA WRITE SIDE (Event Sourcing & Outbox)") specifies `event_store` as the **primary write side** with snapshots for rehydration — i.e., true Event Sourcing. This is not merely "unimplemented," it now **contradicts** ADR-021's explicit choice of versioned-state CQRS over Event Sourcing. Anyone implementing this doc literally today would be building the wrong write model.
- **Asymmetric feature surface risk.** If a Commitment or Habit history feature is ever requested, the natural move is to copy Goal's pattern — but without first deciding whether "durable event log per aggregate" should become a standard, uniformly-applied pattern or stay a Goal-specific choice, it'll be copy-pasted ad hoc a second and third time (same shape of risk Iterations 1-4 already found elsewhere — see synthesis below).

## Technical Debt

- Remove or genuinely use Task's dead `'EventStore'` DI registration (`task.module.ts:117-118`) — either wire a real history feature or delete the unused provider/export.
- Decide and record whether Commitment/Habit should get the same durable-event-log/history pattern as Goal, or whether Goal is intentionally special (e.g., because Goals are long-lived and users most want to see their evolution) — right now this reads as inconsistency, not decision.
- Add expected-version enforcement to the actual state repositories (`InMemoryGoalRepository`, `InMemoryCommitmentRepository`, and presumably Task/Habit's equivalents) so "versioned" repository is true in practice, not just in naming.
- `postgresql_physical_model.md`, `event_store_model.md`, `read_models.md` need a rewrite reflecting versioned-state CQRS as the write model (already flagged as stale in `docs/00-framework/REVIEW_STATUS.md` priority 3 — this iteration adds the specific reason the _write-side_ section needs more than archival: its core assumption is now wrong, not just unbuilt).

## Recommendations

1. **Keep the additive-durable-log pattern** (it's correctly designed) but make a deliberate call on scope: either generalize it to all four aggregates via a shared mechanism, or explicitly document Goal as a special case and stop implying (via dead DI wiring in `task.module.ts`) that Task has the same capability.
2. **Add real optimistic concurrency to the state repositories before any real persistence migration.** This is cheap to do now (in-memory, single process) and expensive to retrofit later once a real DB and real concurrent users exist. Concretely: `save()` should accept/verify an expected version and throw on mismatch, exactly like `InMemoryEventStore` already correctly does — the pattern to copy already exists in this same codebase.
3. **Rewrite the write-side section of `postgresql_physical_model.md`** around versioned-state CQRS (a `commitments`/`goals`/`tasks`/`habits` table each with an `version` column and compare-and-swap update) plus keep a slimmed-down `event_log` table for the Goal-history-style feature specifically — don't keep specifying `event_store`-as-source-of-truth plus `snapshots` for rehydration, since nothing in the actual decided architecture rehydrates from events.
4. **Before building a real Postgres adapter, budget real time for serialization** — this codebase has never done it, so treat it as new work, not "wiring," when scoping that future effort.

## Priority

**Medium.** Nothing is broken today — the one real use case (Goal history) works correctly, and the demo-scale, low-concurrency, single-process reality means the missing optimistic-concurrency enforcement hasn't caused a visible bug yet. But this compounds exactly like Iterations 1-4's Medium findings: every additional feature built on the current repository/event-store shape (a second aggregate's history feature, real Postgres persistence, multi-user concurrent access) inherits an untested concurrency gap and a stale target-architecture doc, and gets more expensive to fix the longer it's built on top of.

## Fase 1 Synthesis

Looking across all five reports: **every abstraction in this codebase is well-shaped but has only ever been exercised by exactly one concrete case, so none of them have been stress-tested by a second real consumer.** Iteration 1's two incompatible `AggregateRoot` base classes never had to reconcile because nothing forced a second usage pattern to confront the first. Iteration 2's repository ports have never been proven with a second real adapter — only the in-memory one exists. Iteration 3's Goal/Commitment/Task boundary was never explicitly decided because Task's direct `goalId` link was the first (and so far only) case that needed an answer, and it got one by default rather than by decision. Iteration 4's redundant idempotency checks exist because the aggregate-owns-idempotency rule was correctly applied in some handlers (`CompleteTaskCommandHandlerCore`) and not enforced consistently as a pattern across all of them. This iteration finds the same shape again: the durable-event-log pattern is real and correct for Goal, but was never generalized or tested against a second aggregate, and the "versioned" repository pattern has never had its version check actually exercised because nothing has yet required concurrent writes to the same aggregate. None of this is breakage — it's **an architecture that has only ever been asked one question at a time**, and every Medium-priority finding in Fase 1 is a version of "what happens when a second, different case shows up" going unanswered.
