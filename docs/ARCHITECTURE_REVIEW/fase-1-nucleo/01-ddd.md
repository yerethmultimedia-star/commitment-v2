# DDD (Aggregates, Value Objects, Domain Events, Ubiquitous Language)

**Scope:** tactical DDD patterns only. Bounded Contexts (strategic design) is Iteration 3 — not covered here.

## Current State

`packages/domain/src/` implements **five real, fully-modeled aggregates** (`Goal`, `Commitment`, `Task`, `Habit`, `Identity`) plus two smaller ones (`Device`, `Reminder`/notifications). All seven extend a generic `AggregateRoot<IdType>` (`packages/domain/src/shared/aggregate-root.ts`), which itself extends `Entity<IdType>` (`shared/entity.ts`) for identity-based equality. Every aggregate:

- Uses a private constructor + static factory (`register`/`create`) that returns the aggregate already carrying its first recorded event.
- Enforces invariants inside instance methods (`activate`, `pause`, `complete`, etc.), not in the application/handler layer, with dedicated typed error classes per violation (e.g. `CommitmentActivationRequirementsNotMetError`, `InvalidCommitmentStateTransitionError` in `commitment/errors/commitment-errors.ts`).
- Emits well-named, past-tense domain events (`CommitmentActivatedEvent`, `GoalHabitLinkedEvent`, `TaskRelinkedToGoalEvent`, etc. — 10 for Commitment, 8 for Goal, 15 for Task, 10 for Habit) carrying minimal, denormalized payloads, not full aggregate snapshots.
- Uses real Value Objects for its core fields: every aggregate has a `value-objects/` folder (`commitment/`, `goal/`, `habit/`, `identity/`, `task/`) with dedicated classes like `CommitmentTitle`, `RecurrencePattern`, `TargetDate`, `SeriesId` — each extending a shared `ValueObject<TProps>` base, validating in the constructor, and exposing only a `.value`/typed getter (verified directly: `commitment/value-objects/commitment-title.ts`, `recurrence-pattern.ts`).

This is **Fully Implemented** as a tactical pattern (not "designed," not "partial") — it is exercised by all seven real aggregates, has passed the codebase's own test suites (`*/aggregate/__tests__` exist per-aggregate), and is the pattern actually used by every backend command handler, confirmed by `apps/backend/src/task/application/preconditions/commitment-activation.preconditions.ts` (see Strengths).

## Strengths

1. **Invariants stay inside the aggregate, even under a real architectural constraint.** `Commitment.activate(hasExecutionPlan: boolean)` needs to know "does this Commitment have at least one linked Task," but Commitment cannot query Task without breaking aggregate-boundary rules. Rather than leaking the check into a handler (the common shortcut), ADR-022 introduces a Port (`CommitmentActivationPreconditionsPort`) implemented by `TaskBasedCommitmentActivationPreconditions` (`apps/backend/src/task/application/preconditions/commitment-activation.preconditions.ts`) that resolves the _fact_ externally — the aggregate still owns the _decision_ and still throws (`CommitmentActivationRequirementsNotMetError`). This is the textbook-correct way to handle a cross-aggregate precondition without a saga; it is real, working code, not a doc describing an intention. Why it matters: this is exactly the kind of decision that's easy to get wrong under deadline pressure (just query the other repository from inside the handler and skip the aggregate check), and this codebase got it right.

2. **"No Meaningless Events" is an enforced rule, not a convention.** `Commitment.rename()`, `.updateDescription()`, `.changePriority()` all diff old vs. new value and return early with no event if nothing changed (explicit comment: `// Rule #77 — No Meaningless Events`, `commitment.ts:250,267,328`). `activate()`/`pause()`/`resume()`/`cancel()`/`complete()` are all idempotent no-ops when already in the target state. Why it matters: event-sourced systems that skip this accumulate noise events that inflate every replay and every read-model projection for no informational gain — this one doesn't.

3. **Value Objects are real, not typedef aliases.** `CommitmentTitle` validates length (1–`MAX_TITLE_LENGTH`) and trims in the constructor; `RecurrencePattern` is a private-constructor VO with a named factory and an `isRecurring()` domain query method, not just an enum wrapper. `UniqueEntityId` (the base for all `*Id` VOs) validates UUID shape and freezes itself. This is Value Objects doing actual domain work, not ceremony.

4. **Domain event metadata already anticipates the Framework's AI-authorship distinction.** `DomainEventMetadata.actorType: 'USER' | 'SYSTEM' | 'AI_PROPOSAL'` (`core/domain-event.interface.ts:8`) is a forward-looking piece of modeling — it's exactly the field you'd need to enforce Chapter 5's axiom ("The AI proposes; it does not enact") at the event-store level, by recording whether a state change was user-enacted or AI-proposed. See Weaknesses for why this doesn't currently do anything.

## Weaknesses

1. **Two incompatible `AggregateRoot` base classes exist in the same package.** `core/aggregate-root.base.ts` (untyped `string` id, `recordEvent` sets `this._version = event.metadata.aggregateVersion` — version comes _from_ the event) is a **separate, parallel implementation** from `shared/aggregate-root.ts` (generic `AggregateRoot<IdType>`, `recordEvent` does `this._version += 1` — version is _self-incremented_, ignoring whatever the event's own metadata says). All seven real aggregates (Identity, Habit, Commitment, Task, Reminder, Goal, Device — verified by grepping every aggregate's import) use `shared/aggregate-root.ts`. The `core/` version has exactly **one** consumer: `appearance/models/appearance.model.ts` — which lives under `models/`, not `aggregate/`, suggesting it may not even be a true DDD Aggregate Root, just a model that reused the base class opportunistically. Why this matters: two versioning semantics for "what does an aggregate's version mean" is a correctness landmine — if `AppearanceModel` is ever wired into real event-sourced persistence (Iteration 5's topic) alongside the other seven, optimistic-concurrency checks based on `version` would behave inconsistently between the two families with no compiler-level warning that they disagree.

2. **Ubiquitous Language governance has drifted badly from the code it's supposed to describe — worse than previously flagged.** `docs/02-domain/UBIQUITOUS_LANGUAGE.md` (v1.1.0, dated 2026-07-04, status "Active" — not part of the already-known-stale batch) defines 11 terms and **does not define `Task` or `Habit` at all** — two of the four core aggregates that exist in code today. It lists `Goal` as an "**Aggregate Candidate**" (not yet confirmed), when Goal is now a fully shipped, real `AggregateRoot`. It defines `Microaction` (`Microacción`) as a core term — the exact concept `docs/PROJECT_AUDIT.md` and the Framework's Appendix already found does not exist in the shipped domain (replaced by the richer `Task`). `docs/02-domain/canonical_dictionary.md` (already known-stale, Junio 2026) is worse still: it defines `GOAL`'s "Aggregate Owner" as **"Commitment Aggregate (Root)"** — i.e., the opposite of the current, real hierarchy where `Goal` sits above `Commitment` (`Goal.linkCommitment`, `goal.commitmentIds`) — and invents a `PROJECT` entity and a tiered `BLUEPRINT` (N1/N2/N3) concept, neither of which has any code representation. Why it matters: "Active, v1.1.0" status on `UBIQUITOUS_LANGUAGE.md` implies a team could reasonably treat it as current — but on the two most-built aggregates (Task, Habit) it says nothing at all, and on Goal it describes a state one refactor behind reality.

3. **The domain's own stated hierarchy is not what the domain actually enforces.** `Goal.ts`'s own header comment states "Goal -> Commitment -> Task/Habit," and ADR-019's diagram repeats this. But `Task` (`task/aggregate/Task.ts`) has both `commitmentId: CommitmentId | null` **and** `goalId: string | null` as independent, nullable fields — a Task can attach directly to a Goal with no Commitment involved at all. This isn't a documentation problem (unlike #2) — it's the aggregates themselves disagreeing with the comment that describes them. Not necessarily wrong as a product decision (direct Goal→Task can be legitimate), but it means the "strict hierarchy" language used in `Goal.ts`, ADR-019, and casual descriptions elsewhere is inaccurate and should either be corrected in the comment or the model should decide to actually enforce the hierarchy it claims.

4. **`Blueprint` — a Level-1 concept in `THE_COMMITMENT_FRAMEWORK.md` Chapter 3 — has no aggregate, no value object, not even a stub.** This was already confirmed at zero hits by exhaustive grep (`docs/PROJECT_AUDIT.md` §2); worth restating here specifically as a DDD gap: there is no tactical modeling exercise that has even begun for the Framework's own stated design-then-commit boundary. Every aggregate here starts life already in a `Draft`/pre-committed state (`Commitment.register()` → `CommitmentState.Draft`, same pattern for `Goal`) — meaning the closest thing to "Blueprint" in the actual domain is just the Draft state of the eventual aggregate, not a separate, freely-discardable object as Chapter 3 describes. If Blueprint is ever built, it is not a trivial addition to the existing model — it's a new upstream stage the existing `Draft` states would need to relate to correctly.

5. **`actorType: 'AI_PROPOSAL'` is dead typing.** Grepped for every other reference to `AI_PROPOSAL` in the domain/backend/mobile code: zero hits beyond the type definition itself. No event constructor ever sets it (all sampled events hardcode `actorType: 'SYSTEM'`, e.g. `CommitmentRegisteredEvent`). This is aspirational schema, not enforced behavior — see Iteration 13 (AI Platform) for the fuller picture, but it's relevant here because it means today's event history cannot actually distinguish a user action from an AI-proposed one at the data level, despite the schema appearing to promise that it can.

## Risks

- **Silent version-semantics mismatch (Weakness 1)** is a real correctness risk _if and when_ `AppearanceModel` (or any future aggregate built against `core/aggregate-root.base.ts`) is wired into a shared event-sourced persistence path alongside the seven aggregates on `shared/aggregate-root.ts`. Low likelihood today (in-memory-only persistence per Iteration 5, and Appearance doesn't look aggregate-shaped), but the risk compounds the longer both base classes coexist unremarked.
- **Language drift compounds decision cost.** Every future conversation about the domain (with a human, or with an AI agent reading these docs cold) that trusts `UBIQUITOUS_LANGUAGE.md` as "Active" will reason from a model missing 2 of 4 core aggregates and misdescribing a third. This is a governance risk more than a code risk, but it's the same category of problem `THE_COMMITMENT_FRAMEWORK.md`'s own Appendix was created to solve for its four prior product docs — this one wasn't caught by that process because it lives in `02-domain/`, not `01-product/`.

## Technical Debt

- Remove or formally justify `core/aggregate-root.base.ts` — either delete it if `appearance.model.ts` doesn't need event-sourcing semantics, or migrate it to `shared/aggregate-root.ts`'s versioning scheme so there is exactly one definition of what "version" means for an aggregate in this codebase.
- `docs/02-domain/UBIQUITOUS_LANGUAGE.md` needs a real rewrite (not archival — unlike `canonical_dictionary.md`, this one's status field claims currency): add `Task` and `Habit`, promote `Goal` out of "Candidate," remove or clearly mark `Microaction` as superseded.
- Reconcile the "Goal -> Commitment -> Task/Habit" hierarchy language (in `Goal.ts`'s comment and ADR-019) against the actual `Task.goalId`-without-`commitmentId` capability — either fix the comment/diagram or decide whether direct Goal→Task should be constrained.

## Recommendations

1. **Resolve the dual `AggregateRoot` base class before Iteration 5 (Event Store) has to reason about persistence uniformly across aggregates** — this is the one finding here with a direct dependency on upcoming review work, so sequencing it first is cheap insurance, not gold-plating.
2. **Rewrite `UBIQUITOUS_LANGUAGE.md` against the current code**, the same way the Framework's own Appendix already did for `01-product/`'s four prior docs — this doc claims "Active" status, so it's actively misleading in a way the already-archived `canonical_dictionary.md` no longer is (a reader who already discounts a doc marked "Congelado" is safer than one trusting a doc marked "Active, v1.1.0").
3. **Do not build `Blueprint` as a bolt-on.** If/when product decides to implement it, treat it explicitly as a new upstream stage relative to each aggregate's existing `Draft` state, not an independent object with no relationship to the state machines that already exist — this avoids a second, competing "not-yet-real" concept sitting alongside `Draft`.

None of these three require touching the tactical pattern itself (aggregates/VOs/events) — that layer earns its strength rating and shouldn't be disturbed for its own sake.

## Priority

**Medium.** Nothing here is on fire — all seven real aggregates enforce their invariants correctly today, and the value-object/event discipline is genuinely above-average for this project's scale. But the dual base-class inconsistency is a landmine that gets more expensive to defuse the longer it's ignored (worse once Event Store work, Iteration 5, starts depending on `version` meaning one thing), and the Ubiquitous Language drift actively misinforms anyone — human or AI — who trusts a document that still claims "Active" status.
