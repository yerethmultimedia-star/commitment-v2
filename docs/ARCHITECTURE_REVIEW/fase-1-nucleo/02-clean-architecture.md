# Clean Architecture (Layering, Dependency Rule, Modularization)

## Current State

`packages/domain` (`packages/domain/package.json`) has **zero framework/HTTP/DB dependencies** — only `uuid` as a runtime dependency, plus `@commitment/config`/`@commitment/shared` as workspace devDependencies. No NestJS, no Express, no persistence library. The dependency rule is respected at the package-manifest level.

Package dependency graph (built from every `packages/*/package.json` and both `apps/*/package.json`): `config` is the universal base (no internal deps); `domain`, `theme-engine`, `localization`, `platform`, `shared`, `api-contracts` depend only on `config`; `design-system` depends on `config`+`localization`+`platform`+`theme-engine`; `backend` depends on `domain`+`shared`; `mobile` depends on `domain`+`design-system`+`api-contracts`+`localization`+`platform`+`theme-engine`. **No circular dependency exists in this graph** — it is a clean DAG.

Backend module internals (spot-checked `commitment` in full, `task`/`goal` partially): `CommitmentsController` (423 lines) dispatches via `CommandBus`/`QueryBus` (`@nestjs/cqrs`) — it never calls a handler directly. `RegisterCommitmentNestjsHandler` (32 lines total) is a thin NestJS adapter that immediately delegates to `RegisterCommitmentCommandHandlerCore`, a framework-agnostic class constructed from Ports (`VersionedCommitmentRepository`, `DomainEventDispatcher`) — the actual business orchestration doesn't touch a single NestJS import.

Mobile layering (spot-checked 4 of 29 files importing `@commitment/domain`): `habits.api.ts` imports `HabitSummary`/`HabitRecurrenceType`, `goal.model.ts`/`demo-goals.repository.ts` import `Milestone`, `coach.tsx` imports `Recommendation`/`RecommendationType` — all type-only imports, not aggregate classes or business logic invoked client-side. This looks like a legitimate shared-type-kernel pattern, not a layering violation, based on this sample (not exhaustively verified for all 29 files).

## Strengths

- **Dependency rule genuinely holds at the domain boundary.** This was verified directly (reading `package.json`, not assuming docs are right) — `packages/domain` cannot reach into NestJS or any infrastructure concern even if a developer tried, since the dependency isn't there to import.
- **No circular package dependencies anywhere in the monorepo** — checked all 9 internal packages' manifests. This is not automatic in a monorepo this size; someone has been disciplined about it.
- **The NestJS-handler-as-thin-adapter pattern is executed well and consistently.** `RegisterCommitmentNestjsHandler` (`apps/backend/src/commitment/nestjs/register-commitment.nestjs-handler.ts:10-32`) contains zero business logic — it exists only to satisfy `@CommandHandler`/DI and hand off to a framework-agnostic `*CommandHandlerCore` class. This is a genuinely above-average level of discipline for a NestJS + CQRS codebase; most real-world examples let business logic leak into the `@CommandHandler` class directly.
- **Controllers correctly stay at the mediator boundary** — `CommitmentsController` uses `CommandBus.execute()`/`QueryBus.execute()` throughout, with DTO validation (`zod`) and domain-error-to-HTTP-exception mapping as its only real responsibilities. This is the correct amount of controller logic, not more.

## Weaknesses

- **Every one of the 4 real aggregates has two competing repository-port definitions, and one of them is dead code.** Confirmed systemic, not isolated:
  - `packages/domain/src/commitment/repositories/commitment.repository.ts` defines `CommitmentRepository` (`save(): Promise<void>`).
  - `apps/backend/src/commitment/application/ports/versioned-commitment-repository.port.ts` defines `VersionedCommitmentRepository` (`save(): Promise<number>` — explicitly "prepares the interface for future Optimistic Concurrency," per its own comment).
  - Every handler constructor, and the only DI binding that exists (`commitment.module.ts:46-47`, `provide: 'CommitmentRepository', useClass: InMemoryCommitmentRepository`), types against `VersionedCommitmentRepository`. A grep for the domain-level `CommitmentRepository` type (not the coincidentally-same-named DI string token) found **zero usages anywhere outside its own defining file.**
  - The identical pattern repeats for Goal (`goal.repository.ts` vs. `versioned-goal-repository.port.ts`), Task (`task-repository.interface.ts` vs. `task-versioned-repository.port.ts`), and Habit (`habit-repository.interface.ts` vs. `habit-versioned-repository.port.ts`).
  - Why this matters: a maintainer reading `packages/domain/src/commitment/repositories/commitment.repository.ts` — the "correct," Clean-Architecture-approved location for a Port — would reasonably believe it's the live contract `InMemoryCommitmentRepository` implements. It isn't. The real, load-bearing contract lives one layer "out," in the backend app, which inverts the dependency rule's intent (the app-layer port had to be invented because the domain-layer one couldn't be changed/wasn't trusted, or was simply forgotten).
- **`packages/application` is not a package.** It has no `package.json` — it is not a pnpm workspace member, confirmed by `grep -r "@commitment/application"` across the entire repo returning zero hits. It contains exactly two files: `docs/register-commitment.md`, `docs/activate-commitment.md`. This is materially different from `docs/PROJECT_AUDIT.md`'s "empty placeholder package" framing — it isn't an empty scaffold for future code, it's stray documentation sitting inside `packages/`, where every sibling directory name implies a real, buildable package.
- **A real cross-module circular dependency exists between `CommitmentModule` and `TaskModule`, currently resolved by relocating a handler rather than resolving the coupling.** `commitment.module.ts`'s own comment states it directly: `ActivateCommitmentNestjsHandler` is registered inside `TaskModule`, not `CommitmentModule`, "because [TaskModule] needs TaskRepository to resolve CommitmentActivationPreconditions... Registering it here would require CommitmentModule to import TaskModule, a circular dependency (TaskModule already imports CommitmentModule)." This is transparently documented (good), but it means the answer to "which module owns Activate-Commitment" is decided by import-cycle avoidance, not by bounded-context ownership — a real signal that Commitment and Task have a two-way dependency the current module boundary doesn't cleanly express.

## Risks

- **The Ports & Adapters abstraction has never been exercised with a second adapter, for any of the 4 aggregates.** Every port — Versioned{Commitment,Goal,Task,Habit}Repository — has exactly one implementation that has ever existed: the `InMemory*` one. This isn't a defect (in-memory is ADR-021's deliberate, reasoned choice), but it means the actual payoff of this layering — swap the adapter, keep the application logic — is completely unverified in practice. If the eventual Postgres implementation surfaces a method the interface doesn't cleanly support (e.g., pagination, transactions spanning multiple aggregates), that will be discovered for the first time then, not now.
- **The duplicate-port pattern will replicate itself.** Any new aggregate added following the existing template will likely copy the same "domain repository interface + backend versioned port" split, because that's what the codebase currently models as the norm — compounding the confusion identified above rather than it being a one-off.

## Technical Debt

- 4 dead domain-level repository interface files (`packages/domain/src/{commitment,goal,task,habit}/repositories/*`) — unused, should be reconciled or deleted.
- `packages/application`'s 2 stray markdown docs — either give the package a real `package.json` and purpose, or move the docs into `docs/` and delete the folder.
- The Commitment↔Task module circular dependency is currently a workaround (handler relocation), not a resolved design decision — worth its own explicit record (an ADR or a note in the existing ADR-022) rather than living only as a code comment.

## Recommendations

1. **Consolidate the two repository-port definitions per aggregate into one, living in `packages/domain`.** The Port belongs in the domain layer per Clean Architecture's own rule (a repository interface is a domain concept, not an application concept) — recommend folding the "Versioned" (optimistic-concurrency-ready `save()` return type) contract into the domain-level interface and deleting the backend-local duplicate, rather than the reverse. This improves Maintainability (one source of truth per Port) and reduces onboarding confusion for anyone reading the domain package expecting it to be authoritative.
2. **Resolve the CommitmentModule/TaskModule circular dependency as an explicit architectural decision**, not a standing workaround — e.g., document it as an addendum to ADR-022 (which already deals with this exact precondition), stating plainly whether this coupling is permanent and acceptable or should be restructured (e.g., a shared "commitment-activation" sub-module) before the module count grows further. Improves Maintainability and reduces the risk that the next engineer "fixes" the circular import by introducing an actual cycle instead of understanding why the handler lives where it does.
3. **Fix `packages/application`.** Either turn it into a real package or delete it and move its docs under `docs/`. Low effort, removes a confusing false signal in the monorepo's structure (Developer Experience).
4. **Do not add new Ports & Adapters ceremony beyond what exists today until at least one port has been proven with a real adapter.** This is a recommendation to _pause_ further abstraction, not to add more — the existing pattern is sound in shape; what's missing is validation, not more layers. When real persistence work starts (already an open question in `docs/00-framework/REVIEW_STATUS.md`), treat the first real adapter as a deliberate test of whether the current interface shapes actually hold, and budget time to revise them if they don't.

## Priority

**Medium.** Nothing here is broken today — the dependency graph is genuinely clean, no cycles exist, and the NestJS-adapter-as-thin-shell discipline is executed well above the norm for this kind of stack. But the systemic (4x, one per aggregate) duplicate Port definitions are an active source of future confusion that compounds with every new aggregate added on the same template, and the fact that Ports & Adapters has never been tested with a second adapter is a real unknown that will surface exactly when persistence work begins — which `docs/00-framework/REVIEW_STATUS.md` already flags as an open, undated question. Nothing here demands stopping current work, but it should be addressed before (not during) the first real-persistence migration.
