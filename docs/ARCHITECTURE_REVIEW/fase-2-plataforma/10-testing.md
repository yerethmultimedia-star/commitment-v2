# Testing Strategy

## Current State

Test volume by layer (counted directly, not estimated): `packages/domain` has 15 spec files covering `commitment`, `goal`, `habit`, `task`, `identity`, `calendar`, `core`, `shared` — notably **zero** spec files for `appearance`, `dashboard`, `insights`, `devices`, `notifications`, `messaging`. `apps/backend` has 22 spec files (unit-level, e.g. `apps/backend/src/commitment/application/commands/pause-commitment.handler.spec.ts`) plus a real e2e suite at `apps/backend/test/` (`app.e2e-spec.ts`, `commitments.e2e-spec.ts`, `goals.e2e-spec.ts`, `jest-e2e.json`). `apps/mobile` has 8 test files.

CI (`.github/workflows/ci.yml`) has two jobs: `backend-ci` (lint, `nest build`, then `pnpm --filter backend run test`) and `mobile-ci` (**typecheck only** — `pnpm --filter mobile exec tsc --noEmit`, no test execution step at all).

## Strengths

- **Domain aggregate tests are genuinely invariant-driven, not happy-path theater.** `packages/domain/src/goal/__tests__/goal.spec.ts` and `commitment/__tests__/commitment.spec.ts` assert real business rules: `Goal.activate()` throwing `GoalActivationRequirementsNotMetError` both for missing description and missing linked Commitment (lines 155, 162), `Commitment.activate()` throwing `CommitmentActivationRequirementsNotMetError` specifically for the ADR-022 execution-plan precondition (line 221, comment cites the ADR directly). This is the review's best-tested layer, proportionate to its importance as the one place all business rules live (Iteration 1's finding).
- **Idempotency is explicitly tested at the aggregate level**, matching the code's own "Rule #77" discipline found in Iteration 1: `commitment.spec.ts` line 177 ("activating an already active commitment is idempotent") and line 294 ("is idempotent when already paused") both assert no-throw and correct state on repeat calls.
- **A real, well-built e2e suite exists** (`apps/backend/test/commitments.e2e-spec.ts`, `goals.e2e-spec.ts`) — this is not one-off Playwright QA-session output (see Weaknesses for that distinction); it's a maintained Jest+Supertest-style suite with its own `jest-e2e.json` config and a `uuid.mock.ts` test helper, suggesting real engineering investment, not an afterthought.

## Weaknesses

- **`apps/mobile/package.json` has no `test` script at all.** The 8 mobile test files that exist (confirmed via `find`) have no npm/pnpm entry point to run them — not "run but not in CI," genuinely unrunnable via any package script today. `mobile-ci` only typechecks.
- **`packages/domain`'s 15 spec files — the best-tested code in the review — are never executed by CI.** `packages/domain/package.json` has a real `test` script (`jest`), but `.github/workflows/ci.yml` never calls it — CI only runs `pnpm --filter backend run test`, which is `apps/backend`'s own Jest config, not `packages/domain`'s. `turbo.json`'s `test` task (`dependsOn: []`) would run it under a root `turbo run test`, but CI doesn't invoke that either. The strongest test suite in the codebase currently provides zero regression protection in CI.
- **The e2e suite exists but never runs in CI.** `apps/backend/package.json` has a distinct `test:e2e` script (`jest --config ./test/jest-e2e.json`); `ci.yml`'s `backend-ci` job only runs the plain `test` script. The three e2e specs are real, checked-in, and completely disconnected from any automated gate.
- **`TECH_DEBT.md` Item 6's 2 pre-existing `tsc --noEmit` errors in backend spec files can never fail CI, structurally, not just today.** `apps/backend/tsconfig.build.json` explicitly excludes `**/*spec.ts` (confirmed by direct read), and the `test` script is plain `jest` — neither the build step nor the test step ever runs `tsc --noEmit` over spec files. This isn't a debt item waiting to be fixed opportunistically; it's permanently invisible to any current CI signal, which is worth knowing before assuming "it's logged, it'll get picked up eventually."
- **Several `TECH_DEBT.md` entries describe Playwright-driven investigations** (e.g. Item 33's Golden Path walkthrough, multiple "confirmed via Playwright" notes across recently-resolved items) **but no `playwright.config.*` file, and no `.spec.ts`/`.e2e.ts` file matching a Playwright pattern, exists anywhere in the repo** (confirmed via repo-wide search). These were real, valuable manual verification sessions — but they are not a regression suite, and nothing prevents the exact bug they found from silently recurring.

## Risks

- **TD-003 (Iteration 4's confirmed redundant-idempotency-check finding) is a concrete, verified example of a class of bug this test suite structurally cannot catch.** `apps/backend/src/commitment/application/__tests__/activate-commitment.handler.spec.ts` line 83 ("should be idempotent — activating three times returns same version with one event total") and `pause-commitment.handler.spec.ts` line 84 ("idempotent when already paused") both explicitly test the exact scenario TD-003 affects — and both pass today, redundant check and all. This confirms the tests are doing their actual job (asserting observable behavior is correct) — but proves that behavior-level tests, however good, cannot and will not flag an architecturally-redundant-but-behaviorally-invisible implementation choice. That requires code review or static analysis, not more tests of this kind.
- **A CI that silently skips its two most substantial test suites (domain unit tests, backend e2e tests) is a worse trust signal than having no CI at all**, because a green checkmark today implies more coverage than actually runs. Anyone trusting "CI is green" as a proxy for "the domain layer's invariants still hold" or "the API contract still works end-to-end" is trusting something that isn't actually being checked.
- **No test exercises concurrent writes**, consistent with Iteration 5's finding of zero optimistic-concurrency enforcement on the real repositories — there is currently no test that could even fail if concurrent-write handling regressed, because the capability itself doesn't exist yet.
- **The Iteration 9 gamification-copy-vs-ADR contradiction (streak language shipped across 3 surfaces) is structurally untestable by anything in this suite** — there is no content-governance/lint mechanism that could flag shipped copy contradicting a governance decision (ADR-006/010). This isn't a gap in this test suite specifically; it's a category of check (content-policy conformance) that doesn't exist in this codebase's tooling at all.

## Technical Debt

- `apps/mobile` needs a `test` script and CI step — currently a pure gap, not a broken thing.
- CI needs to actually invoke `packages/domain`'s test script (either directly or via `turbo run test` at the root, scoped appropriately) — the highest-value, lowest-effort fix in this report, since the tests already exist and are good.
- `test:e2e` needs a CI job (likely needs to run against a real running instance or a test-mode bootstrap — not investigated in depth this pass, flagged for whoever picks this up).
- `TECH_DEBT.md` Item 6 should be re-described accurately: not "2 pre-existing errors to fix opportunistically" but "2 errors invisible to CI by construction until `tsc --noEmit` is added as its own step, independent of build/test."

## Recommendations

1. **Wire `packages/domain`'s existing test suite into CI.** This is close to a pure Quick Win: the tests exist, are good, and the fix is a CI config change, not new test-writing. Recommend flagging as a Quick Win candidate in the Findings Register given the effort/value ratio matches the bar already set by TD-003 and the Design System lint-rule finding.
2. **Add a mobile test-execution step**, even a minimal one — the current state (tests exist, no script, no CI) is strictly worse than either "no tests" (no false expectation) or "tests run" (real protection).
3. **Decide deliberately whether `test:e2e` becomes a CI gate now or is explicitly deferred** — given the app is pre-production and pre-persistence (Iteration 5), there's a legitimate argument this can wait, but it should be a stated decision, not silent omission, especially since the suite already exists and represents real prior investment.
4. **Do not try to "fix" TD-003 by adding more behavior tests** — Iteration 4 already fully diagnosed the fix (delete the redundant handler-level check); this iteration's contribution is confirming that no new test is needed or would help catch this class of issue in the future. The right prevention mechanism is a code-review convention or lint rule ("no state-check in a CommandHandler; trust the aggregate"), not additional test coverage.
5. **Don't invest in Playwright/e2e-browser automation as a first move** — the manual investigation sessions already logged in `TECH_DEBT.md` demonstrate real value from ad hoc Playwright usage; converting the highest-value ones (Golden Path walkthroughs) into a small, maintained suite would capture that value going forward, but this is worth sequencing after the two near-zero-effort CI wiring fixes above.

## Priority

**High.** Not because anything is functionally broken today (all cited tests currently pass), but because this is a reliability-of-signal problem: a passing CI badge currently represents less coverage than it appears to, across the two most substantial suites in the codebase (domain unit tests, backend e2e tests) plus a third area (mobile) with no test execution capability at all. This sits at the same tier as Iteration 6's auth gap and Iteration 9's shipped gamification copy — not "unfinished work," but "a decision or an oversight that silently narrows a safety net everyone else in this review has been implicitly relying on when a report said 'tests exist for X.'"
