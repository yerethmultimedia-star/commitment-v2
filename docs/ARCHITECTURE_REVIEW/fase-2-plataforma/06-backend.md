# Backend

Scope: `apps/backend` as a whole application — composition, cross-cutting concerns, API surface, error handling, auth, health/observability wiring. Domain modeling, CQRS internals, and persistence were already covered in Fase 1 (`fase-1-nucleo/01-ddd.md` through `05-event-store.md`) and are not re-derived here.

## Current State

**Fully Implemented, pre-production.** `app.module.ts` registers a coherent set of modules: `HealthModule, ObservabilityModule, CqrsModule, CommitmentModule, TaskModule, HabitModule, GoalModule, NotificationsModule, DevicesModule, MessagingModule`, plus `ScheduleModule`/`BullModule` (real Redis-backed). `main.ts` bootstraps OpenTelemetry (`otelSDK.start()`) before `NestFactory.create()` — correct ordering, since NestJS's own instrumentation needs to be patched in before any module is instantiated. Global `/v1` prefix (`setGlobalPrefix`), Swagger at `/api/docs`, a global `ProblemDetailsExceptionFilter`. Routes are extensive and real across all 6 domain modules (confirmed live in a prior session, not just by file inspection).

**Zero authentication exists.** Not "not yet wired" in the sense of a stub interface waiting for an implementation — there is no auth package dependency at all (`grep` of `package.json` for `passport`/`jsonwebtoken`/`@nestjs/jwt`/`bcrypt`/`argon2`: zero hits), no `AuthGuard`, no strategy, no middleware performing any check. The one place identity enters the system, `RequestContextMiddleware` (`apps/backend/src/observability/middleware/request-context.middleware.ts:20-22`), says so in its own comment: _"In a real app, identityId would be extracted from the JWT token via an AuthGuard. For now, we leave it undefined or try to extract from a dummy header if it existed."_ It then reads `req.headers['x-identity-id']` verbatim, unauthenticated. On the write side, `identityId` is a client-supplied field in the request body (e.g. `commitments.controller.ts:71`, `z.string().uuid('Invalid identity id UUID format')` — validated for _shape_, not _ownership_).

## Strengths

- **Module composition is coherent and matches the domain boundaries** Iterations 1-5 already validated — no surprise modules, no god-module.
- **Health checks are real, not decorative.** `HealthModule` (`observability/health/health.module.ts`) wires `@nestjs/terminus` to two custom indicators, `OutboxBacklogIndicator` and `ProjectionLagIndicator`, both of which read actual `MessagingModule` state (imported directly into `HealthModule`). Verified live in a prior session: `/v1/health` returned real `pendingCount`/`lag` values, not a hardcoded `{status:'ok'}`. This is above-average rigor for a pre-production backend.
- **The dominant error-handling pattern, where it's applied, is genuinely sound.** Spot-checked `goals.controller.ts`'s `activate()` (line 276-302) and 8 other methods in the same file: each catches the handler's specific `*NotFoundError`/`*StateConflictError` subclasses and rethrows as `NotFoundException`(404)/`ConflictException`(409), with an explicit `BadRequestException`(400) fallback for anything else caught in that block. This is a deliberate, working translation from domain errors (which correctly extend plain `Error`, not `HttpException` — keeping `packages/domain` framework-agnostic, consistent with Iteration 2's Clean Architecture finding) to correct HTTP semantics.
- **`ProblemDetailsExceptionFilter` (RFC 9457) is correctly shaped** — `type`/`title`/`status`/`detail`/`instance`, `application/problem+json` content-type, plus `requestId`/`correlationId` for traceability. Genuinely useful for debugging, not cosmetic.
- **Request correlation is real infrastructure, not an afterthought** — `RequestIdMiddleware` + `RequestContextMiddleware` populate `correlationId`/`causationId`/`requestId` via `RequestContext` (AsyncLocalStorage-based, `@commitment/shared`), consumed by the Pino logger's `customProps` — traceable logs by default.

## Weaknesses

- **No authorization exists anywhere, by construction, not by omission.** Every read filters by whatever `x-identity-id` header the caller sends; every write trusts whatever `identityId` UUID the caller puts in the body. There is no check that the caller _is_ that identity. This isn't "missing an edge case" — the entire request surface has no notion of "this caller may only act as themselves."
- **The error-translation pattern (Strength above) is hand-rolled per method, not centralized.** Spot-checked: the same `catch (error) { if (error instanceof XNotFoundError) throw new NotFoundException(...); if (error instanceof XStateConflictError) throw new ConflictException(...); ... }` shape repeats in every one of `goals.controller.ts`'s ~9 command methods, each importing differently-aliased copies of the same two handler-defined error classes (`GoalNotFoundError as RenameGoalNotFoundError`, `as UpdateGoalDescriptionNotFoundError`, `as ActivateGoalNotFoundError`, etc. — `goals.controller.ts:25-62`). This works today but is pure boilerplate that will be copy-pasted onto every future command handler; a shared exception filter or interceptor keyed on error _base class_ (if one existed) would remove this entirely. Confirmed this pattern is specific to Goal's controller in this pass; not verified whether Commitment/Task/Habit controllers repeat it identically, but the shape is consistent enough with what Iterations 1-5 found to expect the same pattern there.
- **No security middleware at all.** `package.json` has no `helmet`, no `@nestjs/throttler`/rate-limiting, no CSRF protection. `app.enableCors()` (`main.ts:17`) is called with zero options — this is a wide-open CORS policy (any origin, per NestJS/Express defaults when unconfigured), not a scoped one.
- **The uncaught-error fallback path in `ProblemDetailsExceptionFilter` leaks internal detail.** When an exception is a plain `Error` (not an `HttpException` — i.e., anything not explicitly caught and translated by a controller, such as a real bug, a null-pointer, or a future real-database error), lines 50-52 set `detail = exception.message` verbatim and return it to the client with a generic `type: 'about:blank'`, `status: 500`. Today this is low-risk (in-memory repos rarely throw novel errors), but it's the kind of gap that becomes a real information-disclosure vector the moment a real database or external service is introduced (Iteration 5 already flagged persistence as in-memory-only, so this hasn't been exercised yet).
- **`packages/backend`'s `env.config.ts`** (Zod schema, checked in a prior session) defaults `SUPABASE_URL`/`SUPABASE_ANON_KEY` to placeholder strings rather than failing to boot when unset. This is a reasonable choice _today_ (nothing actually uses Supabase yet, per Iteration 5), but as a general pattern, defaulting security-adjacent config to a fake-but-valid value means a genuine misconfiguration in a future real deployment would boot "successfully" and fail silently/insecurely later, rather than failing loud at startup.

## Risks

- **Complete authorization bypass by construction** is the dominant risk: any client that can reach this API can read or mutate any identity's data by simply setting a different `x-identity-id` header or `identityId` body field. Currently contained by the app not being publicly exposed (localhost-only in this session), but the code has zero scaffolding — not even a no-op `AuthGuard` — to build on when that changes.
- **Repeated per-controller-method error-mapping boilerplate** (Weakness above) is a compounding maintainability risk: every new command handler needs its own hand-written catch block, and any inconsistency (a missed case, a wrong status code) will only surface as an incorrect API response, not a compile-time or lint failure.
- **Wide-open CORS + no rate limiting** is low risk today (no public exposure) but is exactly the kind of default that survives unnoticed into a production deploy if nothing forces a review before that happens.

## Technical Debt

- `TECH_DEBT.md` Item 6 (2 pre-existing `tsc --noEmit` errors in `register-commitment.nestjs-handler.spec.ts` and `schedule-reminder-on-queued.handler.spec.ts`) — not re-verified line-by-line this pass, but architecturally this is exactly the kind of debt the missing centralized error-mapping (Weakness above) makes worse over time: hand-rolled mocks are more likely to drift from real handler signatures than a thinner, shared abstraction would be.
- `TECH_DEBT.md` Item 35 (Reminder's `commitmentId`→`sourceId`/`sourceType` generalization left 2 backend spec files broken) — same root cause category: a cross-cutting rename done without a shared contract test, consistent with the "boilerplate over abstraction" pattern found in this iteration.
- `TECH_DEBT.md` Item 40 (`commitmentsApi.create()` never sends `id`/`identityId`, so real-mode registration would 400) is, from this iteration's lens, a symptom of a debatable architectural choice, not just a client bug: requiring the _caller_ to generate and supply both the entity `id` and the `identityId` as request-body UUIDs (`commitments.controller.ts:71` and the `id` field alongside it) only works cleanly once there's a real auth layer establishing which identity a given caller is _allowed_ to supply. Today it's just an unenforced contract; that's fine for a trusted single-client demo, but it's the same root gap as the auth finding, wearing a different hat.

## Recommendations

1. **Introduce a real `AuthModule`/`AuthGuard` before any multi-user or public-facing work begins** — even a minimal JWT-based one. This is the single highest-leverage change available: it closes the authorization-bypass risk and gives `identityId` a trustworthy source, which in turn makes Item 40's "client supplies `identityId`" pattern defensible (the guard would supply/verify it, not the client).
2. **Centralize the domain-error → HTTP-status mapping** (e.g., a shared `@Catch(DomainError)` exception filter keyed on error base classes, or a decorator/interceptor) instead of repeating the same catch block per controller method. This directly reduces the Item 6/35 debt category by removing the hand-rolled surface that drifts.
3. **Add `helmet` and a basic rate limiter (`@nestjs/throttler`)**, and scope CORS to actual known origins once any client other than local dev exists. Low effort, standard practice, currently entirely absent.
4. **Sanitize the uncaught-error fallback path** in `ProblemDetailsExceptionFilter` — return a generic `detail` for non-`HttpException` errors in production (`NODE_ENV === 'production'`), logging the real message server-side via the already-working Pino/correlation-ID infrastructure instead of echoing it to the client.

## Priority

**High.** Unlike every Fase 1 finding (all Medium — well-designed abstractions that simply haven't been tested against a second real-world case yet), the auth gap here is not "designed but unproven" — it's **absent by construction, with no scaffolding at all**, and it is the one finding so far in this review that represents an active, complete bypass of a security boundary rather than an untested one. It doesn't block continuing the review, and it isn't yet exploitable in the current local-only deployment — but it is the first thing that must be addressed before this backend is exposed to anything beyond a single trusted developer's machine.
