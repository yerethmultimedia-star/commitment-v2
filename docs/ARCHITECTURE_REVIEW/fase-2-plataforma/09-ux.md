# UX

## Current State

The shipped UX is judged against the product's own stated philosophy (`docs/00-framework/THE_COMMITMENT_FRAMEWORK.md` Chapters 4/6/9, `docs/01-product/PRODUCT_VISION.md` Core Principles), not generic UX best practice.

i18n structure: only `commitments` and `tasks` get dedicated namespace files (`apps/mobile/src/core/i18n/locales/{en,es}/{commitments,tasks}.json`). Goal, Habit, Coach, Insights, and Dashboard copy all live inside a single 818-line `common.json`. Not a user-facing defect by itself, but it's the likely mechanical reason copy consistency drifts by feature (see Weaknesses).

## Strengths

- **Membrane (Chapter 4) mostly holds.** Actual UI copy is natural-language throughout for the common flows: `actions.pause` → "Pausar", `actions.cancel` → "Cancelar", form labels like `habits.form.fields.title.label` → "Título". A user pausing a Commitment or completing a Task never has to parse domain jargon to act. `tasks.json`'s `form.relatedToCommitment` → "Compromiso" is consistent with the already-accepted ADR-019 amendment (current representation, not an exclusive lock) — not a new violation.
- **Explainability (Chapter 6) is met today, even without real AI.** `apps/mobile/src/app/(tabs)/coach.tsx` renders every recommendation card with a `Title` + `Body` pair sourced from `coach-descriptors.ts`, both visible by default, no "tap to see why" interaction required. Critically, the descriptions cite the actual observable evidence driving the suggestion — e.g. `coach.goalFocus.needsAttention.description` = "{{percent}}% completo — dale un poco de enfoque" (percent is real computed progress, not a generic nudge). This confirms the Framework's own claim that Chapter 6 constrains explainability, not AI-ness — the rule-based Coach (confirmed non-AI, Iteration 6/`docs/PROJECT_AUDIT.md`) already clears this bar.
- **Non-punitive framing holds for Commitment/Task lifecycle transitions specifically.** `commitments.json`'s `workspace.confirm.cancel.description` = 'Cancelar "{{title}}". No podrás reactivarlo.' and `tasks.json`'s `confirm.cancel.description` = '"{{title}}" pasará a Cancelada. Puedes reabrirla más adelante si lo necesitas.' are calm, factual, and — for Task specifically — explicitly reversible-friendly. No shame language anywhere in the pause/cancel/resume strings for these two entities.
- **TECH_DEBT.md Item 8 (invisible submit buttons) is already fixed in code, doc is stale.** Verified directly: `CommitmentForm.tsx:5` and `HabitForm.tsx:5` both now `import { Button } from '@commitment/design-system'`, not raw Tamagui. The primary-task-completion blocker this item described no longer exists.

## Weaknesses

- **Pervasive streak/gamification language directly contradicts two already-decided ADRs and Framework Chapter 9 — this is the most important finding of this iteration.** `common.json` copy is saturated with loss-aversion streak framing across three separate surfaces:
  - Dashboard: `dashboard.widgets.currentStreak.activeMsg` = "¡Mantén la llama encendida!" ("Keep the flame lit!"), `dashboard.hero.streak.title` = "¡Completaste {{count}} tareas esta semana!"
  - Insights: `insights.streakHighlight.title` = "Mejor Racha" ("Best Streak"), `insights.habitConsistency.averageStreak`/`activeStreaks`
  - Coach: `coach.tips.protectStreak.title`/`.description` = "Llevas una racha de {{count}} días" / "Completa una tarea hoy para mantenerla" (protect it), and — most direct — `coach.risks.habitStreaksAtRisk.title`/`.description` = "{{count}} rachas de hábitos en riesgo" / "Complétalas hoy para mantener la racha" (streaks _at risk_, framed as something to defend against loss).
  - This is not a stray widget. `ADR-006` ("Exclusión de Gamificación") and `ADR-010` ("Eliminación de Rachas") already explicitly rejected this mechanic, and Framework Chapter 9 states plainly: _"the temptation to substitute [a completion measure] with a streak... should be treated as exactly that: a temptation, not a solution."_ The "risk of losing a streak" framing is the textbook loss-aversion pattern Chapter 9's "reflection over punishment" principle exists to rule out — it is emotionally the opposite of "reflection," it's anxiety-driven compliance pressure. This was already logged as an unresolved _field-level_ tension (`Habit.currentStreakDays`, `REVIEW_STATUS.md`) — this iteration's contribution is confirming the contradiction is not confined to a data field, it is fully surfaced, repeatedly, in shipped user-facing copy across three different screens.
- **A narrower membrane friction point: achievement-type duality.** `coach.achievements.goalsCompleted` ("{{count}} objetivos completados") and `coach.achievements.commitmentsCompleted` ("{{count}} compromisos completados") are presented as two distinct achievement categories the user sees. To understand why they received one message and not the other, a user has to implicitly grasp that "completing a Goal" and "completing a Commitment" are different countable events — a small but real instance of exactly the vocabulary-distinction Chapter 4 says the product should never require.
- **"Alto contraste" (High Contrast) setting actively lies to the user (Item 28, verified still open).** The toggle animates, persists, and visually confirms a change occurred — but has zero downstream visual effect. For a UX audit specifically (as opposed to a pure a11y compliance check), this is worse than the feature being absent: it's a broken promise the UI itself makes.

## Risks

- **Product-philosophy credibility risk.** The streak-language finding isn't a UX polish gap — it's copy actively working against two decisions the project already made deliberately (ADR-006, ADR-010) and against the Framework chapter that was just formally approved (Chapter 9, `docs/00-framework/REVIEW_STATUS.md`, 2026-07-19). Every day this ships, it's evidence to any future reviewer (human or AI) that the Framework doesn't actually govern the shipped product, undermining the entire point of the governance work done this week.
- **Accessibility as exclusion risk, not just compliance gap.** `PRODUCT_VISION.md`'s own Target Audience includes "anyone who repeatedly abandons personal objectives" — a broad, non-technical population, plausibly including users for whom assistive technology matters more than average. Given that, the confirmed lack of keyboard navigation on web (Item 15) and absence of any formal accessibility audit (Item 7) are a real fit-for-stated-audience risk, not defensible as "fine for an internal demo."

## Technical Debt

- `TECH_DEBT.md` Item 26 (only 1 of 4 Insights stat cards interactive, no affordance cue) and Item 23 (Goal Workspace's "Upcoming" list excludes same-day tasks after midnight) — not re-audited in depth this iteration, but both are real primary-flow-adjacent UX defects consistent with this iteration's "does the primary action work as the user expects" lens; still open per prior findings, no evidence found to contradict that.
- Item 8 should be marked resolved in `TECH_DEBT.md` — confirmed fixed in code (see Strengths).

## Recommendations

1. **Rewrite the streak/achievement copy family to align with Chapter 9 before any further Coach/Insights/Dashboard work builds on it.** Replace "at risk" / "keep the flame lit" framing with language that reflects _evidence of sustained practice_ without implying loss or defense (e.g., describing consistency as an observation, not a countdown). This is a copy-only change (i18n strings), not a data-model change — the underlying `Habit.currentStreakDays` field question (already logged) is separate and can be resolved independently of the copy.
2. **Consolidate `goalsCompleted`/`commitmentsCompleted` into a single achievement framing** (e.g., a unified "completed" count, or explicitly bridge the two with copy that doesn't require the user to already know the distinction) — small effort, closes the one real membrane gap found.
3. **Fix or remove the "Alto contraste" toggle** — a setting that visibly does nothing actively erodes trust in every other setting on that screen.
4. **Split `common.json` into per-feature namespaces (`goals.json`, `habits.json`, `coach.json`, `insights.json`, `dashboard.json`)** matching the existing `commitments`/`tasks` pattern — not urgent, but the current structure is the likely reason copy conventions drift between features (no one file to check against).

## Priority

**High.** Every other Medium finding in this review so far was "well-designed but untested against a second case" (Fase 1's cross-cutting theme) or "absent by construction" (Backend's auth gap). This is different in kind: it's shipped, user-facing content actively contradicting two already-signed-off ADRs and a just-approved Framework chapter, across three separate screens, not a single narrow spot. It doesn't block continuing the review, but it's cheap to fix (copy only) and actively costs product-philosophy credibility every day it isn't.
