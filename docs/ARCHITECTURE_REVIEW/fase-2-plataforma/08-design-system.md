# Design System

## Current State

`packages/design-system` is a real, disciplined Tamagui-based component library: ~35 components/primitives across layout, typography, modal, portal, screens, scroll, interaction, focus, accessibility, keyboard, and pickers, plus a genuine token layer (`tokens/{colors,space,motion,themes,typography}.ts`) merged into a single `createTamagui()` config (`tamagui.config.ts`). Dependencies are clean: `@commitment/localization`, `@commitment/platform`, `@commitment/theme-engine`, Tamagui/Gorhom/RN libraries — no dependency on `apps/mobile`, confirming Iteration 2's "no circular dependencies" finding holds specifically for this package too.

Of the 7 previously-logged `TECH_DEBT.md` items scoped to this package, verified current state:

- **Item 9 (no Badge/Chip primitive) — actually RESOLVED in code**, not just planned. `src/components/Badge.tsx` exists: tone/variant/size/shape props, enforced i18nKey-or-label discriminated union, WCAG-verified token pairings reused from `Button`. Its own header comment cites Item 9 by name as the bug it fixes. **`TECH_DEBT.md` was never updated** — same stale-doc-vs-fixed-code pattern Iteration 7 found for Item 40.
- **Item 44/25 (hand-rolled chip-tab pattern) — genuinely resolved**, not a narrow patch. `src/components/ChoiceGroup.tsx` generalizes over `isSelected`/`onSelect`/`labelFor` (not naive `value === option` equality — needed for `DurationInput`'s non-value "Custom…" chip), and is properly wired to the shared interaction system (`useInteractionState`, `FocusRing`, `useHapticBehavior`, `useInteractionAnimation`) that the hand-rolled versions bypassed.
- **Item 12 (`Card`'s unfrozen prop surface) — confirmed still open.** `CardProps extends YStackProps` (`Card.tsx:7`), and `{...props}` spreads onto the root `View` _after_ all fixed defaults (`padding="$4"`, `borderRadius="$4"`, shadow values) at `Card.tsx:122` — any caller can silently override them. No other audited component (`Badge`, `ChoiceGroup`) repeats this pattern; `Card` is the outlier.
- **Item 5 (`HeroCardStrategy` dead code) — confirmed still open and confirmed fully dead**: `apps/mobile/src/features/dashboard/ui/components/strategies/HeroCardStrategy.ts` has zero import sites anywhere in `apps/mobile/src` (verified via repo-wide grep).
- **Item 17 (`GoalProgressBar`/`CircularProgress` → `ProgressMetric` migration) — confirmed still pending**: `GoalProgressInsight.tsx:7` still imports the local `@/features/goals/components/GoalProgressBar`, not the shared `ProgressMetric` from `@commitment/design-system`.
- **Item 20 (Portal-rendered modals ignore the active Experience Theme) — root cause now identified precisely**, see Weaknesses below. This is worse than "ignores theme" as previously logged — it's a structural React-tree placement bug, not a missing-prop oversight.
- **Item 14 (snapshot tests don't exercise per-theme output)** — not independently re-verified this pass beyond the already-logged casing-mismatch root cause; treated as still-open per existing log.

## Strengths

- **A real, consistent component design philosophy**, not just a folder of unrelated components. `Badge`, `Card`, and `ChoiceGroup` all: (a) enforce i18n discipline via a discriminated `i18nKey`-xor-`label` union (never silently allow un-translated raw strings except through an explicit, documented escape hatch), (b) route all interactive state through the same shared primitives (`useInteractionState`/`FocusRing`/`useHapticBehavior`/`useInteractionAnimation`), and (c) use semantic tone tokens (`$accent`/`$success`/`$danger`/etc.) rather than hardcoded colors. This is exactly the kind of cross-component discipline a design system exists to enforce, and it's real, not aspirational.
- **A genuine token layer**, not values that merely happen to look consistent. `tamagui.config.ts` explicitly merges Commitment-specific `space`/`size`/`themes`/motion presets on top of Tamagui's own base config (`@tamagui/config/v3`), with a documented reason for _how_ (the `motion.ts`/`motion.native.ts` split exists because `createAnimations()`'s closure can't be patched after creation — a real constraint correctly worked around, not guessed at).
- **Clean package boundary.** Zero reverse dependency on `apps/mobile`; only depends on other foundational packages (`localization`, `platform`, `theme-engine`). This package could be extracted or reused independently without unwinding app-specific coupling.
- **When TECH_DEBT items here do get fixed, the fixes are real generalizations, not spot patches** — `ChoiceGroup`'s generic `isSelected`/`onSelect`/`labelFor` signature is genuinely more reusable than the 4 call sites it replaced, and its own comment documents why (the non-value "Custom…" chip case).

## Weaknesses

- **Root cause of Item 20, found this pass:** `apps/mobile/src/app/_layout.tsx` renders `<PortalProvider>` wrapping `<AppearanceGate>` (line 73 wraps line 77+), and `PortalProvider`'s own implementation (`PortalContext.tsx:42-47`) renders portaled content (`PortalHost`) as a **sibling** of `{children}`, not a descendant of it. The actual per-user Experience Theme (Midnight/Forest/Sunrise) is applied via `<Theme name={activeAppearance.settings.themeId}>` inside `AppearanceProvider.tsx:113` — nested _inside_ `AppearanceGate`, i.e. inside `PortalProvider`'s `children`, but **not** inside `PortalHost`. Since `PortalHost` sits at the same tree level as `children` (both are direct children of `PortalContext.Provider`), anything rendered through `Portal` (which is how `ModalPrimitive` renders on web — `ModalPrimitive.tsx:97`) structurally escapes the `<Theme name={themeId}>` boundary entirely. It only ever inherits `TamaguiProvider`'s own `defaultTheme` (`_layout.tsx:69` — hardcoded to `'dark'`/`'light'` based on system color scheme, never one of the three custom Experience Themes). This is not a missing style or a forgotten prop — it's a React-tree structural placement issue, and it affects every component built on `ModalPrimitive` (`Dialog`, `BottomSheet`, `ConfirmationDialog`, `ActionSheet`) uniformly, on web specifically (native uses RN's own `Modal`, a different code path with different theming behavior not verified this pass).
- **Zero automated enforcement of design-system adoption.** The root `.eslintrc.json` is generic (`eslint:recommended` + `@typescript-eslint/recommended`, plus `no-console`/`no-unused-vars`) — there is no `no-restricted-imports` rule or equivalent anywhere in the repo blocking `import { Button } from 'tamagui'` in application code (verified: no `.eslintrc*`/`eslint.config*` file anywhere in the repo references `tamagui` or `no-restricted-imports`). This is the single mechanical cause behind essentially all 7 adoption-debt items above — nothing fails a build or a PR check when a developer bypasses the design system, so consistency depends entirely on individual awareness of what already exists.
- **`Card`'s unfrozen prop surface (Item 12) is a real API design defect**, not just a style nit — it silently defeats the whole point of having fixed defaults, and nothing else in the audited surface (`Badge`, `ChoiceGroup`) repeats this mistake, which suggests it's fixable by matching the pattern already used elsewhere in the same package rather than needing a new pattern invented.

## Risks

- **The theming escape (Item 20) will recur for any future Portal-based primitive**, not just the four already affected, because the root cause is structural (where `PortalProvider` sits relative to `AppearanceProvider`), not per-component. Adding a fifth modal-like primitive without first fixing the placement will inherit the same bug automatically.
- **Adoption debt compounds silently.** Because nothing enforces it, every new feature built without checking "does a design-system primitive already exist for this" has equal odds of creating an 8th hand-rolled duplicate, regardless of how many of the first 7 get fixed. Fixing the 7 known instances treats symptoms; the lint-enforcement gap is the actual recurring risk.
- **Untested theme parity on native.** This audit verified the Portal/theme-escape mechanism specifically for web (`Platform.OS === 'web'` branch in `ModalPrimitive.tsx`). Native uses RN's own `Modal` component instead, a different rendering path — whether the same Experience Theme actually reaches native modals correctly, worse, or by luck, was not verified this pass.

## Technical Debt

Confirmed still open (with root causes now attached where found this pass): Item 12 (`Card` unfrozen props), Item 5 (`HeroCardStrategy` dead code, zero call sites), Item 17 (`GoalProgressInsight` still on local `GoalProgressBar`), Item 20 (Portal/Theme structural escape — root cause now precisely identified, see Weaknesses). Confirmed resolved-in-code-but-not-in-docs: Item 9 (Badge exists and is good), Item 44/25 (ChoiceGroup exists and is a real generalization, not a patch) — `TECH_DEBT.md` should be updated to reflect both as closed.

## Recommendations

1. **Fix the `PortalProvider`/`AppearanceProvider` ordering** so portaled content renders as a descendant of the active `<Theme name={themeId}>` boundary, not a sibling of the subtree containing it — this is a placement fix (where these providers nest relative to each other), not a redesign of either provider. Justification: closes Item 20 at its actual root cause rather than patching each of the 4 affected components individually, and prevents the same bug in any future Portal-based primitive. (Maintainability, Product Quality.)
2. **Add a `no-restricted-imports` ESLint rule** (or equivalent) blocking direct `tamagui` component imports from `apps/mobile` feature code, with the design-system package itself exempted. Justification: this is the one mechanical change that would have prevented essentially all 7 already-logged adoption-debt items, and prevents the 8th one for free going forward — much higher leverage than fixing instances one at a time. (Maintainability, Developer Experience.)
3. **Freeze `Card`'s prop surface** by moving `{...props}` before the fixed style props (or splitting `CardProps` into an explicit allow-list instead of extending full `YStackProps`), matching the discipline already present in `Badge`/`ChoiceGroup`. Justification: closes a real defeats-its-own-purpose bug using a pattern that already exists elsewhere in the same package — no new convention needs to be invented. (Maintainability, Product Quality.)
4. **Update `TECH_DEBT.md`** to mark Item 9 and Item 44/25 as resolved (they already are, in code) — low effort, prevents future audits/contributors from re-investigating already-solved problems. (Developer Experience.)
5. Do **not** treat this package's overall architecture as needing a redesign — the token system, i18n discipline, and interaction-state wiring are genuinely sound and should be the template new components follow, not something to rework.

## Priority

**Medium.** Nothing here is broken for the single-user, current-scale product today, and several items already fixed themselves correctly without intervention (Items 9, 44/25). But Item 20's root cause is a real, structural defect (not cosmetic) affecting every current and future Portal-based primitive on web, and the lint-enforcement gap is the one fix in this entire audit phase so far that would prevent an entire _class_ of future findings rather than closing one instance — both are worth prioritizing before Fase 2 closes, even though nothing here rises to the Backend iteration's "zero auth" High-priority bar.
