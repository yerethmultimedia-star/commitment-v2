## Iteration 5 — Event Store & Read Models (2026-07-20)

- **Correction to the finding this iteration started from:** the event store is not dead system-wide. It's dead for `Task` (registered in `task.module.ts:117-118`, zero call sites) but genuinely real for `Goal`: 8 `saveEvents()` call sites across Goal's command handlers, plus a real `getEvents()`-backed read via `get-goal-history.handler.ts` feeding a routed `GET /goals/:id/history` endpoint. Neither Commitment nor Habit register or use it at all.
- Confirmed this is a deliberate, correctly-designed **additive audit log**, not Event Sourcing — `activate-goal.handler.ts`'s own comment states it plainly ("Durable event log (ADR-021) — additive, does not affect step 3's source of truth"), and `applyEvent` is never invoked during aggregate load, only at write time. The actual source of truth is `InMemoryGoalRepository`/`InMemoryCommitmentRepository` holding live object references directly.
- Found a real gap: **no optimistic concurrency enforcement exists on the actual state repositories** — `save()` never checks an expected version before overwriting. The only enforced version check lives in the non-authoritative Goal event log (`InMemoryEventStore.saveEvents`, which does this correctly). A concurrent double-write to real state would silently last-write-win today.
- Assessed the 3 legacy domain docs' write-side design directly: `postgresql_physical_model.md` specifies `event_store` as the **primary write side** with snapshots for rehydration — true Event Sourcing. This doesn't just predate ADR-021, it now actively **contradicts** it. These docs need a rewrite of their write-side model, not just an archival decision.
- Score: Event Store 58/100. Priority: Medium.

### Fase 1 synthesis

Every abstraction found across Iterations 1-5 is well-shaped but has only ever been exercised by exactly one concrete case: two `AggregateRoot` base classes never had to reconcile (Iter 1), repository ports were never proven with a second real adapter (Iter 2), the Goal/Commitment/Task boundary was never explicitly decided because only one case (Task→Goal) ever forced the question (Iter 3), idempotency ownership was applied correctly in one handler but not enforced as a pattern everywhere (Iter 4), and the durable-event-log pattern is correct for Goal but was never generalized or given real concurrency enforcement because nothing has yet required a second aggregate or concurrent writes (Iter 5). None of Fase 1's findings are active breakage — they're all the same shape of gap: **an architecture that has only ever been asked one question at a time, with every "what happens with a second case" left unanswered.**
